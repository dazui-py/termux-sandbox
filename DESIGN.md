# termux-sandbox

`termux-sandbox` creates isolated Termux-like sandboxes using PRoot.

The goal is not to run a separate Linux distribution. The goal is to create independent Termux environments where each sandbox has its own:

- `HOME`
- `PREFIX`
- `TMPDIR`
- package database
- shell configuration
- installed binaries
- project workspace
- logs
- policies and grants

The real Termux installation stays as the host. Each sandbox gets its own Termux-like filesystem under:

```text
~/.termux-sandbox/boxes/<name>/rootfs/
```

Inside the sandbox, programs still see the normal Termux paths:

```
/data/data/com.termux/files/home
/data/data/com.termux/files/usr
```

However, those paths point to the sandbox rootfs, not to the real Termux installation.

## How it works

The sandbox is started with PRoot.

PRoot allows us to emulate a chroot/bind-mount style environment in user space, without requiring root. This makes it suitable for creating isolated Termux environments on normal Android devices.

A sandbox will have its own Termux rootfs, its own home directory, and its own package space.

Installing packages, editing shell configs, or writing files inside the sandbox should not modify the real Termux environment.

## Project split

The project is split into two parts:

```
termux-rootfs    -> clean Termux rootfs builder/cache/source
termux-sandbox   -> sandbox manager/runtime
```

termux-sandbox does not create the base Termux rootfs directly.

The rootfs comes from my separate repository:

https://github.com/dazui-py/termux-rootfs

termux-rootfs is responsible for preparing clean Termux rootfs artifacts from the official Termux bootstrap sources.

For now, rootfs artifacts are built locally. Later, GitHub Actions will build and publish them automatically through workflow runs and GitHub Releases.

## Rootfs source

The sandbox rootfs will not come from random Linux distributions.

termux-rootfs will:

1. detect the target architecture
2. check if the architecture is available in the latest Termux bootstrap release
3. download the official Termux bootstrap
4. cache the downloaded bootstrap
5. avoid redownloading if the cached bootstrap is already up to date
6. extract it into a Termux-compatible layout
7. restore bootstrap symlinks
8. create the required directory structure
9. generate a manifest
10. pack everything into a .tar.zst artifact
11. generate SHA256SUMS
12. publish the artifacts through GitHub Releases later

The output from termux-rootfs will look like this:

```
termux-rootfs-aarch64-<bootstrap-tag>.tar.zst
manifest-aarch64-<bootstrap-tag>.json
SHA256SUMS-aarch64-<bootstrap-tag>
```

When a new sandbox is created, termux-sandbox will use a rootfs artifact generated by termux-rootfs as the base filesystem.

At first, termux-sandbox will use locally built rootfs artifacts.

Later, it will be able to:

1. detect the device architecture
2. check its local rootfs cache
3. download a matching artifact from:
   https://github.com/dazui-py/termux-rootfs/releases
4. verify the artifact using SHA256SUMS
5. extract it into:
   `~/.termux-sandbox/boxes/<name>/`
6. use the extracted rootfs as the base filesystem for the sandbox

## Sandbox storage layout

All sandboxes will be stored under:

```
~/.termux-sandbox/
```

The structure will be:

```
~/.termux-sandbox/
├── boxes/
│   └── <name>/
│       ├── rootfs/
│       ├── metadata.conf
│       ├── grants.conf
│       ├── policy.conf
│       └── logs/
├── cache/
│   └── rootfs/
└── config/
```

Each sandbox will have its own rootfs:

```
~/.termux-sandbox/boxes/<name>/rootfs/
```

Inside that rootfs, the Termux layout will exist here:

```
rootfs/data/data/com.termux/files/home
rootfs/data/data/com.termux/files/usr
```

So inside the sandbox, programs will see:

```
/data/data/com.termux/files/home
/data/data/com.termux/files/usr
```

But those paths will belong to the sandbox, not to the real Termux.

## Core commands

Planned commands:

```
termux-sandbox create <name>
termux-sandbox enter <name>
termux-sandbox run <name> <command>
termux-sandbox list
termux-sandbox info <name>
termux-sandbox destroy <name>
```

## Creating a sandbox

Command:

```
termux-sandbox create <name>
```

What it will do:

1. validate the sandbox name
2. check if the sandbox already exists
3. detect the device architecture
4. look for a matching rootfs artifact in cache
5. if missing, use a locally built rootfs artifact for now
6. later, download the matching artifact from termux-rootfs releases
7. verify the artifact with SHA256SUMS
8. create `~/.termux-sandbox/boxes/<name>/`
9. extract the rootfs into that box
10. write metadata.conf
11. write the selected policy profile
12. create an empty grants.conf
13. create logs/

Example metadata:

```
name=test
arch=aarch64
profile=dev
created_at=2026-05-04T22:00:00Z
rootfs_source=termux-rootfs-aarch64-bootstrap-xxxx.tar.zst
```

At this point, the sandbox exists but is not running yet.

## Entering a sandbox

Command:

```
termux-sandbox enter <name>
```

What it will do:

1. check if the sandbox exists
2. read metadata.conf
3. read policy.conf
4. read grants.conf
5. build the PRoot command
6. start a shell inside the sandbox

The PRoot command will use the sandbox rootfs as `/`.

The sandbox environment will be started with clean variables:

```
HOME=/data/data/com.termux/files/home
PREFIX=/data/data/com.termux/files/usr
TMPDIR=/data/data/com.termux/files/usr/tmp
PATH=/data/data/com.termux/files/usr/bin:/data/data/com.termux/files/usr/bin/applets
SHELL=/data/data/com.termux/files/usr/bin/bash
```

The goal is that inside the sandbox, this works:

```bash
echo "$HOME"
echo "$PREFIX"
which bash
pkg --version
```

And it should behave like a normal Termux environment.

## Running a command inside a sandbox

Command:

```
termux-sandbox run <name> <command>
```

Example:

```
termux-sandbox run test pkg install git
```

What it will do:

1. check if the sandbox exists
2. prepare the same PRoot environment used by enter
3. execute the given command
4. exit after the command finishes
5. write logs

This is useful for scripts and automation.

Example:

```
termux-sandbox run test ./install.sh
```

If the script installs packages, edits shell configs, or writes into `$PREFIX`, those changes stay inside the sandbox.

## Listing sandboxes

Command:

```
termux-sandbox list
```

What it will do:

1. scan `~/.termux-sandbox/boxes/`
2. read each metadata.conf
3. print name, arch, profile and creation date

Example output:

```
NAME        ARCH      PROFILE    CREATED
test        aarch64   dev        2026-05-04
buildbox    aarch64   strict     2026-05-04
```

## Showing sandbox info

Command:

```
termux-sandbox info <name>
```

What it will show:

- name
- architecture
- profile
- rootfs source
- created date
- rootfs size
- grants
- logs path

Example:

```
Name: test
Arch: aarch64
Profile: dev
Rootfs: bootstrap-2026.05.03-r1+apt.android-7
Path: ~/.termux-sandbox/boxes/test
Grants: none
```

## Destroying a sandbox

Command:

```
termux-sandbox destroy <name>
```

What it will do:

1. check if the sandbox exists
2. show the full path that will be deleted
3. require the user to type the sandbox name
4. delete the sandbox directory

It must not use a simple y/N confirmation.

Correct confirmation:

```
Type the sandbox name to confirm:
```

This avoids deleting the wrong box by accident.

## Initial profiles

The project will start with two profiles.

### strict

The strict profile is intended for maximum separation from the real Termux environment.

Rules:

- no /sdcard
- no real HOME
- no real PREFIX
- no external mounts by default
- deny filesystem egress by default
- logs enabled

### dev

The dev profile is intended for normal project testing.

Rules:

- isolated HOME
- isolated PREFIX
- optional /workspace mount
- logs enabled
- external access only through explicit grants

The default profile should be:

**dev**

Because most users will want to test projects, not run a locked-down prison.

## Grants

Grants are explicit permissions given to a sandbox.

Example future command:

```
termux-sandbox grant test ~/projects/myrepo /workspace --rw
```

This means:

- host path: `~/projects/myrepo`
- sandbox path: `/workspace`
- mode: read-write

The grant will be stored in:

```
~/.termux-sandbox/boxes/test/grants.conf
```

Example:

```
/home/user/projects/myrepo:/workspace:rw
```

When entering the sandbox, termux-sandbox will read this file and add the correct PRoot bind.

Without grants, the sandbox should not see real host directories.

## Egress supervision

The main security idea is filesystem egress supervision.

The sandbox can freely operate inside its own rootfs. When something tries to access resources outside the sandbox rootfs, that is treated as an egress attempt.

The basic rule is:

```
sandbox rootfs -> allowed
outside sandbox rootfs -> egress
```

Examples of egress events:

- reading outside the sandbox rootfs
- writing outside the sandbox rootfs
- executing a binary outside the sandbox rootfs
- accessing /sdcard
- accessing host paths
- modifying granted workspace paths

Future policy decisions may include:

- ALLOW
- DENY
- ASK_ONCE
- ASK_ALWAYS
- LOG_ONLY
- KILL

The initial versions will focus on clean filesystem isolation and explicit grants.

In the early versions, egress will mainly be prevented by not mounting sensitive host paths.

Later, a runtime can detect and supervise egress more deeply.

## Root/su scope

Root and su support are out of scope for the initial release.

This project is focused on normal non-root Termux sandboxing.

Root support may become a separate experimental mode later, but it will not be part of the core MVP.

For now:

- no root mode
- no su integration
- no KernelSU/Magisk/APatch handling

## Implementation plan

The first versions will be written mostly in shell.

The early implementation is mainly orchestration:

- create sandbox directories
- extract rootfs artifacts
- launch PRoot
- manage metadata
- manage policies
- manage logs
- manage grants
- destroy sandboxes safely

Later, a C runtime may be added and compiled with clang.

That runtime will not be a real kernel. It will be a sandbox supervisor/policy runtime responsible for detecting egress attempts, such as access outside the sandbox rootfs.

Possible future runtime structure:

```
runtime/
├── main.c
├── policy.c
├── path_guard.c
└── log.c
```

Possible build command:

```bash
clang runtime/*.c -O2 -Wall -Wextra -o termux-sandbox-runtime
```

The runtime may eventually classify actions like:

- ALLOW
- DENY
- ASK_ONCE
- ASK_ALWAYS
- LOG_ONLY
- KILL

But this is not part of the first MVP.

## First MVP

The first MVP is only this:

- create
- enter
- run
- list
- info
- destroy

The MVP is successful when this works:

```bash
termux-sandbox create test
termux-sandbox enter test
pkg install git
echo "hello" > ~/.test
exit
```

Then outside the sandbox:

```bash
ls ~/.test
```

This should fail.

But this should work:

```bash
ls ~/.termux-sandbox/boxes/test/rootfs/data/data/com.termux/files/home/.test
```

That proves the sandbox has its own home.
